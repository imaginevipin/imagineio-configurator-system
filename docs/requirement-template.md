# Configurator Requirement Parsing Guide

**Version:** 2.0
**System:** imagine.io Configurator Generation System
**Audience:** Claude Code (this document is machine context, not a human-facing form)
**Last Updated:** March 2026

---

## 1. Purpose

This document tells Claude Code how to interpret any configurator requirement it receives. Requirements will arrive in unpredictable formats: plain text paragraphs, structured briefs, Figma links, screenshots, spreadsheets, or a mix of all of these. Claude Code must extract the necessary information, identify gaps, ask clarifying questions, and only proceed to generation once all critical inputs are resolved.

**Cardinal Rule:** Never invent product data. If a piece of information is missing and cannot be reasonably inferred, ask the user. Do not guess option names, prices, colors, or product details.

---

## 2. Parsing Workflow

When Claude Code receives a configurator requirement, follow these phases in order:

```
Phase 1: IDENTIFY THE DESIGN SYSTEM SOURCE
  → Is this PDS (default) or a client design system?
  → If client: what reference material was provided?
  → Extract or request design tokens before anything else.

Phase 2: EXTRACT PRODUCT DATA
  → Product name, category, description
  → Option groups and their options
  → Pricing information
  → Visual assets

Phase 3: DETERMINE LAYOUT AND BEHAVIOR
  → Archetype selection (explicit or inferred)
  → Context (standalone, embed, mobile)
  → Commerce elements (CTA, pricing display)

Phase 4: IDENTIFY GAPS AND ASK QUESTIONS
  → Compare extracted data against the Required Fields list (Section 4)
  → Ask all clarifying questions in a single message, grouped logically
  → Do not ask questions one at a time across multiple messages

Phase 5: GENERATE
  → Only after all critical gaps are resolved
  → Follow the Configurator Architecture Reference for layout and components
  → Follow the resolved design system for all visual tokens
```

## 3. Design System Resolution

This is Phase 1 and it must happen before anything else. The design system determines every color, font, spacing value, border radius, and visual treatment in the configurator.

### 3.1 How the Configurator Consumes Design Values

Every configurator generated by Claude Code uses a flat set of CSS custom properties (variables). These variables have semantic names like `--surface-primary-default` or `--text-default-heading`. The configurator's HTML and CSS reference ONLY these semantic variable names, never raw hex codes.

Where the actual values for these variables come from is a separate concern:

- **PDS mode:** Values come from the Prism Design System's three-tier architecture (Brand > Alias > Component). Claude Code loads `prism-token-structure.md` and uses its predefined mappings.
- **Client mode:** There is no three-tier system. There may not be any design system at all. Claude Code must look at whatever reference material exists (a screenshot, a URL, a PDF, a Figma file, or just a verbal description) and construct the flat variable set by extracting and deriving visual values directly.

**The configurator's code is identical in both modes.** The only thing that changes is the `:root {}` block where the CSS variables are defined. Think of it as: the configurator is a template with empty color/font/radius slots, and design system resolution is the process of filling those slots.

### 3.2 Mode A: Prism Design System (PDS)

Use PDS when:
- The requirement does not mention any client brand, website, or design system
- The requirement explicitly says "use PDS" or "use our design system" or "imagine.io style"
- The requirement is for an internal demo or showcase

When using PDS, Claude Code should load the full token set from the `prism-token-structure.md` document attached to this project. The CSS `:root` block from the Architecture Reference Section 5.6 applies as-is. No extraction needed. No user confirmation needed.

### 3.3 Mode B: Client Visual Identity (No Design System Assumed)

Use client mode when:
- The requirement mentions a specific brand or client name
- A screenshot of a website or app is provided
- A Figma link to the client's designs is provided
- A document with brand guidelines is provided
- The requirement says "match their website" or "follow their branding" or similar

**Critical assumption in Mode B:** The client probably does NOT have a formal design system. They have a website, or a brand, or a "look." Claude Code's job is to reverse-engineer a consistent set of visual values from whatever is provided. This is not about finding their token file. This is about looking at their visual output and deriving what the tokens should be.

### 3.4 Client Extraction Process

#### Step 1: Identify the Reference Material

| What is Provided | Extraction Approach |
|------------------|-------------------|
| Screenshot of website/app | Visual analysis (see 3.4.1) |
| Figma link | Use Figma tools to extract exact values (see 3.4.2) |
| Brand guidelines PDF/doc | Read and map values (see 3.4.3) |
| Just a URL | Ask user for a screenshot, or use web_fetch if available (see 3.4.4) |
| Verbal description ("blue and clean") | Ask for more specifics, propose a palette (see 3.4.5) |
| Nothing (but client name mentioned) | Ask: "Can you share a screenshot of [client]'s website or any brand reference I can use?" |

#### 3.4.1 Extraction from Screenshot

This is the most common scenario. The user shares a screenshot and says "match this."

**What to extract by observation:**

1. **Primary action color** — Look at buttons, links, active/selected elements, CTAs. This is the single most important color. Everything else derives from it.
2. **Page background** — Usually white or very light grey.
3. **Card/surface background** — The background of content cards, panels, containers. Often matches page background or is slightly different.
4. **Text colors** — Look at headings (usually near-black), body text (usually dark grey), and secondary/caption text (usually medium grey).
5. **Border treatment** — Are borders visible? What color? Are they subtle (light grey) or prominent? What's the border radius: sharp (0px to 2px), moderate (4px to 8px), or rounded (12px to 16px+)?
6. **Font** — Can you identify the typeface? If it's recognizable (Inter, Roboto, Poppins, Montserrat, etc.), name it. If not, describe the category (geometric sans-serif, humanist sans-serif, serif, monospace) and choose the closest common match.
7. **Button style** — Background color, text color, border radius, approximate height, font weight. Is it filled, outlined, or ghost?
8. **Selection indicator** — If the screenshot shows any selected/active state, how is it indicated? Colored border, filled background, checkmark, underline, bold text?
9. **Spacing feel** — Is the layout tight and dense, moderate, or generous and airy? This determines whether to use the standard PDS spacing scale or adjust.

#### 3.4.2 Extraction from Figma

1. Use `get_screenshot` on relevant screens/components to see the visual design
2. Use `get_variable_defs` to check if the file has defined variables/tokens (some will, most won't)
3. Use `get_design_context` on key components (buttons, input fields, cards, navigation) to extract exact CSS values including colors, fonts, sizes, radii
4. If the Figma file has defined variables, use them directly
5. If not, extract values from the components themselves

#### 3.4.3 Extraction from Brand Guidelines Document

1. Look for: color palette (primary, secondary, accent, neutrals), typography (font names, sizes, weights), spacing/grid rules, border/corner radius rules, button/component specs
2. Map whatever is provided to the configurator variable set
3. For anything not specified in the guidelines, derive or use PDS defaults

#### 3.4.4 Extraction from URL

1. Prefer: ask the user to share a screenshot for the most accurate extraction
2. Alternative: if web_fetch is available, fetch the page and analyze visible CSS patterns
3. Look at the computed styles of key elements: buttons, headings, body text, backgrounds, borders

#### 3.4.5 Extraction from Verbal Description

This is the weakest source. Claude Code should push for more:

```
"To match [client]'s visual style accurately, it would help to have a screenshot of their website or app. Could you share one? 

If not, I can work with what you've described, but I'll need to make some assumptions about specific colors, fonts, and styling that I'll run by you before building."
```

If the user insists on verbal only, Claude Code should propose a complete palette and get explicit approval.

### 3.5 Deriving a Complete Variable Set from Limited Input

In most client cases, Claude Code will extract maybe 5 to 8 values directly. It needs to derive the rest. Here is how:

#### From one primary color, generate the full primary scale:

Given a primary color (e.g., `#2b5c8a`):
- `primary-default`: The color itself (`#2b5c8a`)
- `primary-hover`: 10% to 15% darker. Mix with black or reduce lightness in HSL.
- `primary-subtle`: Very light tint. Set lightness to 92% to 96% in HSL while keeping hue and a touch of saturation.
- `primary-subtle-hover`: Slightly less light. Lightness 86% to 90%.
- `primary-on-color`: Text color on primary background. Usually white (`#ffffff`). If primary is light, use dark text.

**Method (HSL approach):**
```
Primary: hsl(H, S, L)
Hover: hsl(H, S, L - 10%)
Subtle: hsl(H, S * 0.3, 95%)
Subtle-hover: hsl(H, S * 0.4, 90%)
```

#### From screenshot backgrounds, derive the neutral scale:

- `surface-default`: Card background (usually `#ffffff`)
- `surface-page`: Page background (usually `#ffffff` or `#fafafa`)
- `surface-page-secondary`: Slightly darker than page (usually `#f5f5f5` to `#f8f8f8`)
- `surface-disabled`: Medium-light grey (`#d4d4d4` to `#e0e0e0`)

#### From screenshot text, derive the text scale:

- `text-heading`: Darkest text visible (usually `#111111` to `#1a1a1a`)
- `text-body`: Body text (usually `#333333` to `#555555`)
- `text-caption`: Lighter secondary text (usually `#666666` to `#888888`)
- `text-placeholder`: Lightest text (usually `#999999` to `#aaaaaa`)
- `text-disabled`: Similar to placeholder or slightly lighter

#### From screenshot borders, derive border values:

- `border-default`: Subtle borders (usually `#e5e5e5` to `#f0f0f0`)
- `border-secondary`: Slightly more visible borders (usually `#d0d0d0` to `#e0e0e0`)
- `border-primary`: Same as primary color (for selected states)
- `border-focus`: Same as primary color (for focus rings)

#### Border radius:

Observe the client's buttons and cards:
- **Sharp:** 0px to 2px → set `--br-100: 2px; --br-200: 4px`
- **Moderate:** 4px to 8px → set `--br-100: 4px; --br-200: 8px` (PDS default)
- **Rounded:** 8px to 12px → set `--br-100: 8px; --br-200: 12px`
- **Pill/Full:** 16px+ or fully round → set `--br-100: 12px; --br-200: 20px`

#### What always stays from PDS (unless client explicitly overrides):

- Spacing scale (`--s-100` through `--s-1400`): These are structural, not brand. Keep PDS values.
- Z-index scale: Always PDS.
- Transition timing: Always PDS.
- Layout proportions and breakpoints: Always from Architecture Reference.
- Typography SIZE scale (H1 through caption sizes): Keep PDS scale unless client provides specific sizes. The font FAMILY changes, but the size hierarchy stays.

### 3.6 The Configurator Variable Set

This is the flat set of CSS variables that every configurator uses. Claude Code populates these values from either PDS (directly) or client extraction (derived). The variable NAMES never change. Only the VALUES do.

```css

  --surface-primary-default-hover:    /* Button hover bg */;
  --surface-primary-default-subtle:   /* Light tint bg (option hover, tags) */;
  --surface-primary-default-subtle-hover: /* Slightly deeper tint */;

  --text-primary-default-hover:       /* Primary text hover */;

  --icons-primary-on-color:           /* Icon on primary bg (usually white) */;

  --s-400: 16px;
```

### 3.7 Design System Confirmation

After resolving the variable set (Mode B only), Claude Code must present the values to the user in plain language and ask for confirmation BEFORE generating any configurator code:

```
"Based on [the screenshot / Figma file / brand document] you shared, here are the visual values I'll use for the configurator:

Brand Color: #2b5c8a (from the buttons on the site)
Brand Hover: #1e4a73 (derived, 12% darker)
Backgrounds: white (#ffffff) with light grey (#f5f7fa) as secondary
Text: near-black (#1a1a2e) for headings, dark grey (#4a4a5a) for body
Borders: light grey (#e2e8f0), 1px
Corners: moderately rounded (8px)
Font: Inter (identified from the site, falling back to system sans-serif)
Selection indicator: 2px colored border (matching brand color)
Spacing and layout: using standard configurator spacing scale

Does this look right? Anything you'd like me to adjust?"
```

The user may:
- Approve as-is → proceed to generation
- Correct specific values → update and proceed
- Provide additional reference material → re-extract

### 3.8 What Stays Constant Across Both Modes

| Concern | Source | Changes per client? |
|---------|--------|-------------------|
| Layout archetypes and proportions | Architecture Reference | No |
| Component behavior (states, interactions) | Architecture Reference | No |
| Responsive breakpoints and transformations | Architecture Reference | No |
| Spacing scale | PDS (Architecture Reference Section 5) | No (unless client explicitly provides one) |
| Z-index hierarchy | Architecture Reference Section 5.4 | No |
| Transition timing | Architecture Reference Section 5.5 | No |
| Typography SIZE scale (px values) | PDS | No (unless client provides specific sizes) |
| Colors, font family, border radius | PDS or Client extraction | **Yes** |
| Selection indicator style | Architecture Reference (border-based) | May change if client uses a different pattern (fill, checkmark, etc.) |
| Button style (filled vs outlined) | Architecture Reference (filled primary) | May change if client uses outlined or ghost buttons |

This section defines what information Claude Code needs and how to extract it from loose requirements.

### 4.1 Critical (Must Have — Ask If Missing)

| Field | What to Look For | Question to Ask If Missing |
|-------|-----------------|---------------------------|

## 5. Extraction Patterns

Claude Code should recognize these patterns in loose requirements and map them to structured data.

### 5.1 Recognizing Option Groups

Look for these patterns in the text:

**Explicit listing:**
> "The chair comes in 3 fabrics: Linen, Velvet, and Leather"
→ Group: "Fabric", Type: material (or text pill if no images), Options: [Linen, Velvet, Leather]

**Comma-separated in parentheses:**
> "Available colors (Red, Blue, Green, Black, White)"
→ Group: "Color", Type: color, Options: [Red, Blue, Green, Black, White]

**Key-value style:**
> "Material: Oak, Walnut, Cherry / Finish: Matte, Gloss, Satin"
→ Two groups: "Material" (type: material) and "Finish" (type: material or text pill)

**Implied from product description:**
> "Users should be able to customize the leg color and seat fabric"
→ Two groups: "Leg Color" (type: color) and "Seat Fabric" (type: material). Options unknown — ask.

**Table or spreadsheet format:**
> | Color | Hex | Price |
> | Red | #cc0000 | +$0 |
> | Blue | #0000cc | +$25 |
→ Group: "Color", Type: color, Options with hex values and price modifiers

### 5.2 Recognizing Dependencies

Look for:
- "only available with..."
- "when X is selected, show Y..."
- "X options change based on Y"
- "not compatible with..."
- "requires..."
- "if leather, then no light colors"

### 5.3 Recognizing Layout Preferences

| User Says | Map To |
|-----------|--------|
| "side by side", "split view" | SBS-LR or SBS-RL |
| "full screen preview", "big image" | FW-BP or FW-OP |
| "embed", "widget", "fits inside the page" | COMPACT |
| "mobile only", "phone", "responsive" | VSTACK |
| "step by step", "wizard", "guided" | WIZARD |
| "build your own", "modular", "assemble" | BUILD |
| "luxury", "immersive", "premium" | SPLIT |
| "three column", "show specs" | TC |

### 5.4 Recognizing Design System Signals

| User Says | Interpretation |
|-----------|---------------|
| "Use our design system" / "PDS" / "Prism" / "imagine.io style" | Mode A: PDS |
| "Match their website" / "Follow [brand] design" / client name mentioned | Mode B: Client. Ask for reference material if not provided. |
| "Here is a screenshot of their site" / image attached | Mode B: Screenshot extraction |
| "Here is the Figma file" / Figma URL shared | Mode B: Figma extraction |
| "Clean, modern" / "minimalist" / vague aesthetic words | Ask: "Do you have a specific website or brand reference I can match? Or should I use PDS as the base?" |
| No design mention at all | Default to Mode A: PDS |

## 6. Clarifying Questions Strategy

### 6.1 Rules

- Ask all questions in one message, not spread across multiple turns
- Group questions logically (product data, then design, then behavior)
- For each question, explain WHY the information is needed (briefly)
- Offer reasonable defaults where possible so the user can just say "yes, use the defaults"
- If the requirement is very complete, do not ask questions just for the sake of it. Proceed.

### 6.2 Question Template

When gaps are identified, structure the clarifying message like this:

```
I have enough to get started, but a few things would make the configurator more accurate:

**Product Data:**
1. [Question about missing options/details]
2. [Question about missing options/details]

**Design:**
3. [Question about design system / visual reference]

**Behavior:**
4. [Question about pricing/CTA/interactions]

If you would like, I can proceed with these defaults and you can adjust later:
- [Default 1]
- [Default 2]
- [Default 3]
```

### 6.3 When NOT to Ask

Do not ask about:
- Spacing values (use architecture defaults)
- Responsive behavior (use architecture defaults)
- Component sizing (use architecture defaults based on option count)
- Z-index values (use architecture defaults)
- Transition timing (use architecture defaults)
- Typography scale (use PDS or client font at standard scale)

These are all handled by the Architecture Reference. Only ask about product-specific data that cannot be inferred.

## 8. Image Data Structure

After extraction, organize image data into this structure:

```yaml
images:
  mode: "static"                     # static | per-angle | per-combination
  
  # Mode: static (single image, no changes)
  static_url: "product.jpg"
  aspect_ratio: "4:3"

  # Mode: per-angle (multiple views, not affected by selections)
  views:
    - id: "front"
      label: "Front"
      url: "front.jpg"
      is_default: true
    - id: "side"
      label: "Side"
      url: "side.jpg"

  # Mode: per-combination (images change based on selections)
  image_triggers: ["group_1", "group_2"]   # which groups affect the image
  combinations:
    - match:
        group_1: "charcoal-grey"
        group_2: "cotton"
      views:
        - id: "front"
          label: "Front"
          url: "grey-cotton-front.jpg"
          is_default: true

  # If no images provided:
  placeholder: true
```

## 10. Design System Data Structure

The resolved design system. This is always a flat set of values, regardless of source.

```yaml
design_system:
  mode: "pds"                        # pds | client
  source: "prism-token-structure.md" # or "screenshot", "figma", "brand-doc", "url", "verbal"
  confirmed: true                    # has the user confirmed these values? (always true for PDS, must be true for client before generation)

  # For PDS mode: these are the PDS defaults. No need to list them; just reference the Architecture doc Section 5.6.
  # For client mode: these are the extracted/derived values.

  tokens:
    primary: "#ec4e0b"               # PDS default. Client mode: extracted from buttons/CTAs
    primary_hover: "#bd3e09"         # PDS default. Client mode: derived darker shade
```

## 12. Validation Before Generation

Before writing any code, Claude Code must verify:

1. **Design system is resolved:** Either PDS is confirmed as default, or client tokens are extracted and confirmed by user
2. **At least one option group exists** with at least two options
3. **Each option group has a type** (explicitly stated or inferred)
4. **CTA is defined** (label at minimum)
5. **Archetype is selected** (explicitly or via decision tree)
6. **No placeholder product data exists** that should be real data (e.g., option names are not "Option 1", "Option 2" unless the user explicitly wants placeholders)

If any of these fail, ask the user before proceeding.
